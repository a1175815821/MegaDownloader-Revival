# MegaDownloader Revival v2.0 深度审计结论与整改计划

## 一、总体结论

### 审计结果

- **Release 构建成功**：已使用 Visual Studio 2022 Build Tools 对 `Release|x86` 执行完整 Rebuild，生成了 `I:\v1.9\bin\Release\MegaDownloader.exe`。
- **不建议将当前版本作为稳定版发布**。核心下载路径存在文件写出目录逃逸、解压路径穿越、下载完整性未校验、断点恢复可能“假完成”等高风险问题。
- 核心 MEGA 下载功能值得保留，但需要先完成一次以**路径安全、数据完整性和生命周期管理**为中心的重构。
- Remote Web、Streaming Library、ELC、Steganography 等周边功能的安全和维护成本明显偏高。如果没有明确活跃用户和维护资源，建议删除，而不是继续修补。
- 当前工程最大的问题不是“不能编译”，而是存在大量**能正常编译、正常显示成功状态，但实际行为错误**的隐蔽缺陷。

### 发布判断

| 判断项 | 结论 |
|---|---|
| 能否编译 | 可以 |
| 能否作为内部测试版使用 | 修复 P0 后可以 |
| 能否作为公开稳定版发布 | 当前不可以 |
| 是否可能损坏或覆盖文件 | 是 |
| 是否可能把损坏文件标记为成功 | 是 |

| 推荐动作 | 冻结正式发布，先完成 P0/P1 整改 |

## 二、普通用户视角：最可能遇到的问题

1. **下载显示完成，但文件实际上损坏**
   - 当前只执行 AES-CTR 解密，没有验证 MEGA 文件 MAC。
   - 网络截断、Range 错位、旧 `.part` 文件、断点 XML 错误都可能产出损坏文件，而界面仍显示完成。

2. **恢复下载后得到全零文件或旧文件**
   - 当断点 XML 表示所有分块已完成，但 `.part` 文件不存在时，程序可能创建一个同长度的空文件并将其重命名为最终文件。
   - `.part` 长度正确但内容属于其他文件时，也缺少文件身份验证。

3. **文件夹下载写出用户选择的目录**
   - MEGA 文件夹中的远端名称未经严格路径过滤。
   - 恶意名称可能包含 `..`、路径分隔符、根路径或 Windows 设备名。

4. **自动解压覆盖下载目录外的文件**
   - 压缩包条目可以通过 `../` 或绝对路径逃出目标目录。
   - 压缩炸弹还可能耗尽磁盘空间。

5. **网络异常时下载永久卡住**
   - 服务器提前结束响应时，读取循环可能不断得到 0 字节但不退出。
   - 用户会看到速度为零、进度不动且任务无法自然失败。

6. **大于约 34 GiB 的文件后半段解密失败**
   - CTR counter 使用 32 位整数，偏移超过约 34.36 GiB 后可能溢出。
   - 大文件续传尤其容易触发。

7. **VLC 播放能开始，但拖动、续播或末尾播放失败**
   - Streaming 的 HTTP Range 实现不符合标准。
   - 无 Range 请求也返回 206，suffix range、越界和末块处理均存在错误。

8. **设置或下载队列意外恢复默认**
   - 配置和队列保存不是可靠的原子提交。
   - 保存中断后，主文件可能消失；部分情况下虽然存在 `.bak`，启动时也不会恢复。
   - 用户不会收到明确的恢复提示。

9. **“完成后关机”在存在失败任务时仍可能触发**
   - 当前更接近“队列停止活动后关机”，而不一定是“全部成功后关机”。
   - 应用缺少清晰可见的倒计时取消入口。

10. **语言界面缺字、出现空文本或回退到错误语言**
    - 不同语言包键数量差异较大。
    - 当前语言缺少键时，没有稳定回退到完整的 `en-US`。
    - 自定义语言文件还可能被内置文件覆盖。

11. **第二次打开程序传入的链接偶尔丢失或粘连**
    - 单实例通信使用追加文本文件且不写换行。
    - 快速连续启动可能把多条消息连接成一条无法解析的数据。

12. **设置过大的缓冲区后程序内存暴涨或崩溃**
    - Buffer、Package、连接数和速度限制缺少模型层的安全上限。
    - 手工修改配置文件可以绕过部分 UI 限制。

#### P0 修复要求

- 建立统一的 `PathGuard`，所有写入、移动、重命名、解压和删除均必须通过。
- 对候选路径执行 `Path.GetFullPath`，再验证其位于 canonical 下载根目录之内。

  - `..`；

- 删除操作必须使用恢复时保存的 canonical 最终路径，不能临时重新拼接未经验证的字段。

### P1：数据完整性、安全、死锁及严重可靠性问题

#### 1. 未验证 MEGA 文件 MAC

- 位置：
  - `I:\v1.9\Clases\Criptografia.vb:310-320`
  - `I:\v1.9\Clases\FileDownloader.vb:886-977`
- 问题：
  - 当前仅执行 CTR 解密；
  - MEGA key 中的 MetaMAC 没有参与最终完整性校验。
- 影响：
  - 网络损坏、Range 返回错误、磁盘旧数据、恶意响应都可能静默通过。
- 决策：**修复**。
- 验收要求：
  - MAC 不符时不得重命名为最终文件；
  - 状态必须为“完整性校验失败”；
  - 保留或隔离 `.part` 供诊断，但不能标记成功。

#### 2. HTTP Range 响应验证不足

- 位置：`I:\v1.9\Clases\FileDownloader.vb:821-869`
- 问题：
  - 没有强制要求合法的 206 响应；
  - 未完整验证 `Content-Range` 的起始、结束和文件总长度。
- 影响：
  - 服务器忽略 Range 返回完整文件时，数据可能写入错误偏移；
  - 代理或 CDN 错误响应可能造成静默损坏。
- 决策：**修复**。
- 要求：
  - 分块请求必须收到 206；
  - 返回区间必须与请求完全一致；
  - 总长度必须在整个任务生命周期内保持一致；
  - 200、416、区间错位或缺失 `Content-Range` 均进入受控重试或失败。

#### 3. 提前 EOF 导致无限空读

- 位置：`I:\v1.9\Clases\FileDownloader.vb:907-1032`，重点 `943-951`
- 问题：`Read()` 返回 0 后没有进度，外层循环仍继续。
- 用户表现：任务永久停在某一进度，速度为零。
- 决策：**修复**。
- 要求：在尚未读取预期长度时收到 0 字节，应立即作为 `UnexpectedEndOfStream` 失败并进入有限重试。

#### 4. 断点元数据可制造“假完成”

- 位置：
  - `I:\v1.9\Clases\Fichero.vb:772-788`
  - `I:\v1.9\Clases\FileDownloader.vb:558-598,630-651`
- 问题：
  - 直接信任 `StartIndex`、`Size`、`Index`、`AllFinished`；
  - 不校验分块重叠、缺口、越界和总长度；
  - `.part` 缺失时仍可能保留完成状态。
- 用户表现：恢复后得到空文件、旧文件或不完整文件。
- 决策：**修复**。
- 要求：
  - `.part` 不存在时全部进度归零；
  - 分块区间必须完整覆盖 `[0, fileLength)` 且互不重叠；
  - 元数据必须绑定链接标识、节点 ID、文件长度及加密 key 指纹；
  - 最终成功以 MAC 验证为准，而不是 `AllFinished=True`。

#### 5. CTR counter 在大文件偏移处溢出

- 位置：
  - `I:\v1.9\Clases\FileDownloader.vb:895-897`
  - `I:\v1.9\Clases\Criptografia.vb:651-674`
- 问题：
  - `CInt(Math.Ceiling(ChunkStart / blockSize))` 将 counter 强制转换为 `Int32`；
  - 超过约 34.36 GiB 的偏移可能溢出；
  - 非 16 字节对齐偏移没有正确处理 block 内 offset。
- 决策：**修复**。
- 要求：
  - counter seek 使用 `Int64` 或 `UInt64`；
  - 使用 `blockIndex = offset \ 16`；
  - 使用 `inBlockOffset = offset Mod 16`；
  - 不对齐时先生成并丢弃 block 内对应字节；
  - Range 起点与请求不一致时直接失败。

#### 6. Remote Web 与 Streaming Library 存在 localhost CSRF

- 位置：
  - `I:\v1.9\HttpModule\WebInterfaceModule.vb:206-220,354-366`
  - `I:\v1.9\HttpModule\StreamingLibraryModule.vb:215-393`
- 问题：
  - 停止、恢复、删除、保存、导入、登出、启动 VLC 等状态修改由 GET 触发；
  - 浏览器中的恶意网页可能向 localhost 发起请求。
- 决策：
  - Remote Web 有明确用户：**重构修复**；
  - 无明确用户：**删除整个功能**。
- 修复要求：
  - 所有状态修改使用 POST；
  - 加 CSRF token；
  - 校验 `Origin`、`Host`；
  - 保持 loopback-only 绑定；
  - 设置合理的认证失败和会话过期策略。

#### 7. Streaming HTTP Range 不符合规范

- 位置：`I:\v1.9\HttpModule\StreamingModule.vb:121,173-179,207-255`
- 问题：
  - 普通请求也返回 206；
  - suffix range、越界、末块和内容长度处理错误。
- 用户表现：VLC 拖动失败、播放中断、尾部无法播放。
- 决策：
  - 有活跃使用者：**重写**；
  - 无活跃使用者：**删除 Streaming**。
- 修复时必须支持：
  - 无 Range：200；
  - 单区间 Range：206；
  - 不可满足区间：416；
  - 正确的 `Content-Range`、`Content-Length` 和 `Accept-Ranges`；
  - 客户端中途断开时取消上游下载。

#### 8. Streaming URL 和 VLC 启动参数注入

#### 9. 自动解压缺少资源配额

## 五、需要统一调整的内部接口和状态模型

1. **下载结果不能继续使用模糊布尔状态**
   - 建议建立明确状态：
     - `Queued`
     - `Preparing`
     - `Downloading`
     - `Verifying`
     - `Downloaded`
     - `Extracting`
     - `Completed`
     - `Cancelled`
     - `NetworkFailed`
     - `IntegrityFailed`
     - `DiskFailed`
     - `ExtractionFailed`
   - 下载完成和解压完成必须分离。

2. **断点元数据增加版本和文件身份字段**
   - 至少包含：
     - schema version；
     - MEGA node/file identifier；
     - 文件总长度；
     - key 指纹；
     - chunk 区间和已持久化长度；
     - checkpoint 时间。
   - 不兼容或无法验证的旧元数据应安全归零，而不是猜测恢复。

3. **统一路径安全接口**
   - 文件创建、重命名、解压和删除只接受经过验证的 canonical path。
   - 禁止各模块自行使用 `Path.Combine` 拼接远端输入。

4. **统一网络请求策略**
   - 下载、更新、URL 导入、Steganography、ELC 等不得各自定义无限等待请求。
   - 统一超时、取消、代理、重试分类和日志脱敏。

5. **配置验证下沉**
   - UI 和 XML 加载均调用同一套配置模型校验。
   - 手工编辑配置不能绕过连接数、内存、端口和速度上限。

## 七、必须通过的测试和验收场景

### 路径与文件安全

- MEGA 文件名分别包含：
  - `../`、`..\`；
  - 绝对路径；
  - UNC 路径；
  - `CON`、`NUL`、`COM1`；
  - 尾点、尾空格；
  - 极长路径；
  - 仅大小写不同的冲突名称。
- 解压包含 Zip Slip 条目的 ZIP、RAR、7z。
- 删除任务时篡改配置中的本地路径。
- 验收：任何写入或删除都不得逃出 canonical 下载根目录。

### 下载和完整性

- 服务器返回 200 而不是 206。
- `Content-Range` 起点、终点或总长度错误。
- 响应读取中途返回 EOF。
- 连接重置、超时、代理错误和 416。
- 模拟超过 34 GiB 的 CTR 偏移。
- 模拟非 16 字节对齐偏移。
- 文件内容 MAC 不匹配。
- 零长度文件。
- 验收：损坏数据绝不进入完成状态。

### 断点恢复

- XML 表示全部完成，但 `.part` 不存在。
- `.part` 长度正确但属于另一个文件。
- chunk 重叠、缺口、越界。
- 服务端文件长度变化。
- key 或 node identifier 变化。
- 保存 checkpoint 时强制终止进程。
- 验收：无法证明有效的部分必须重新下载，不允许猜测完成。

### 并发和关闭

- 下载启动期间立即关闭程序。
- 校验、解压、Streaming 和 Web 请求同时进行时关闭。
- 多次点击停止、继续和删除。
- worker 抛异常时验证 Mutex 释放。
- 验收：退出后不得继续写文件或联网，不得留下永久锁和假运行状态。

### 配置与队列

- 保存配置时模拟磁盘满、拒绝访问和进程终止。
- 主文件损坏、主文件缺失、`.bak` 有效。
- 队列保存失败后再次触发相同内容保存。
- 验收：有效 `.bak` 必须自动恢复，并向用户明确提示。

### Web 和 Streaming

- 外部网页向 localhost 发起 GET/POST 请求。
- 缺少或伪造 CSRF token。
- 错误 `Origin` 和恶意 `Host`。
- VLC 执行普通请求、open-ended range、suffix range、越界 range 和末尾 range。
- 验收：跨站请求不能修改状态；Range 响应符合 HTTP 规范。

### UI、语言和平台

- 对全部语言包执行 key 集合一致性检查。
- 125%、150%、200% DPI 和多显示器切换。
- Windows 明暗主题运行时变化。
- x86、x64 或 AnyCPU 的平台行为与发布名称一致。
- 验收：无空文本、关键控件截断、异常主题状态或平台误标。

## 八、最终验收标准

公开发布前必须满足：

- 所有 P0 问题关闭并有自动化回归测试。
- P1 中的路径、MAC、Range、EOF、断点恢复、CTR 溢出、持久化和线程取消全部关闭。
- 不再存在 UI 可触发的 `NotImplementedException`。
- 下载成功必须同时满足：
  1. 收到预期全部字节；
  2. Range 区间验证通过；
  3. 写盘成功；
  4. MEGA MAC 验证通过；
  5. 原子重命名成功。
- Release 流程在 clean checkout 中自动完成构建和测试。
- 安装包不再经过 MPRESS。
- 发布资产提供 SHA-256；有条件时提供 Authenticode 签名。
- 所有保留的周边功能必须有明确维护者和自动化覆盖，否则删除。

## 九、审计边界

本次结论属于：

- 深度静态代码审计；
- 工程、依赖、配置、持久化和发布流程审查；
- VS2022 `Release|x86` 实际 Rebuild 验证。

本次尚未覆盖：

- 真实 MEGA 账号和不同地区节点的完整网络矩阵；
- VLC 多版本动态兼容测试；
- 全部 ZIP/RAR/7z 变体及恶意压缩样本动态利用；
- 多 DPI、多显示器的实际人工交互回归；
- 杀毒软件误报矩阵；
- 长时间高并发、断网恢复和磁盘故障压力测试。

因此当前结论已足以判定**不应直接发布稳定版**；完成 P0/P1 后，还需要一轮动态集成测试和恶意输入验证才能形成最终发布结论。
